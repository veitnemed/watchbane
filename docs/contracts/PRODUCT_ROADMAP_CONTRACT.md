## 0. Что делает продукт (канон)

**Выбранный контур продукта (вариант X, 2026-07-19):**  
Watchbane — **inbox новых рекомендаций**: помогает разбирать неизвестные тайтлы и вести личные списки, а **не** «выбрать один фильм на сегодняшний вечер».

**Одной фразой:** локальное Windows-приложение — **конечная колода кандидатов** (фильм/сериал) с постерами; пользователь сортирует их в **смотрел / сохранить / скрыть**.

**Делает (Task 1 — inbox):**

1. показывает конечную колоду кандидатов **с постерами** (до 10);
2. помогает увидеть свежие / ещё не разобранные предложения из локального пула (TMDb под капотом);
3. принимает решения: **смотрел** (уже видел / знаю) · **сохранить** (в список на потом) · **скрыть** (больше не предлагать);
4. хранит данные **локально** (local-first);
5. учитывает watched / saved / hidden при следующих колодах.

**Не делает (сейчас и до отдельного решения):**

- стриминг и плеер;
- бесконечный каталог / поисковик кино;
- соцсеть, сайт аниме;
- сценарий «Сегодня»: три варианта → кнопка «Смотреть / выбрал» (это **Task 2 / V0** — **parking**, не активный фокус);
- NL / LLM как главную фичу;
- ежедневную настройку фильтров ради простого разбора колоды.

Коллекция (Моё) — место, куда складывается результат разбора (watched / saved / hidden), не замена колоды.

**Честное описание после фазы C:**  
«Watchbane помогает разбирать новые рекомендации и поддерживать личный список кандидатов» — **да**.  
«Watchbane выбирает, что смотреть сегодня» — **нет** (пока не открыт отдельный контур V0).

## 0.1. Продуктовая гипотеза (C0-01)

**Для кого:** сначала для автора Watchbane — человека, у которого регулярно появляются новые или ещё не разобранные фильмы и сериалы.

**Проблема:** интересные кандидаты смешиваются с уже просмотренным и нерелевантным; чтобы решить, что оставить на потом, приходится снова листать большой каталог, помнить прошлые решения или повторно оценивать тот же тайтл.

**Гипотеза:** если Watchbane показывает небольшую колоду до 10 кандидатов с постерами и даёт три понятных решения — **смотрел / сохранить / скрыть**, — автор сможет за несколько минут превратить неопределённость в актуальные личные списки без бесконечного каталога и будет возвращаться к этому разбору.

**Ожидаемый результат сессии:** пользователь знает, какие тайтлы уже знакомы, какие сохранены на потом и какие больше не нужно предлагать. Watchbane не обязан выбрать один фильм для просмотра сейчас.

**Альтернатива, которую заменяет продукт:** ручной поиск и бесконечный просмотр каталога, память о прошлых решениях или разрозненные списки без обратной связи на следующие рекомендации.

Проверка гипотезы и условия её опровержения фиксируются отдельными задачами после рабочего happy path; этот ID определяет только проблему и предполагаемую ценность.

## 0.2. Проверка продуктовой гипотезы (C0-02)

S1–S6 проверяют, что inbox технически и UX-но работоспособен. Они **не доказывают**, что он нужен пользователю. Гипотеза из C0-01 считается подтверждённой только после C3, если автор провёл **5 сессий** минимум в **3 разные дни** и одновременно:

1. открыл Watchbane для собственного разбора, а не только ради проверки задачи;
2. в каждой сессии уверенно разобрал не менее 3 карточек;
3. среди ранее неизвестных тайтлов нашёл кандидатов, которые хотел сохранить или изучить дальше;
4. хотя бы один сохранённый тайтл позднее был открыт из «Моё» как осмысленный результат разбора.

**Опровержение:** если эти условия не выполняются после рабочего C1–C3, не открывать A/B/V0 и не начинать C4 автоматически. Зафиксировать, что именно не сработало: качество выдачи, понятность inbox или отсутствие повода возвращаться; затем пересмотреть гипотезу отдельным решением.

## 0.3. Строгая задача фазы C

> Сделать **inbox-колоду** понятной и быстрой:  
> **открыл → разумное время → колода с постерами → разобрал ≥ 3 карточки** (смотрел / сохранить / скрыть),  
> без обязательных детальных фильтров и без технической цепочки «пул / replenish» на экране.

Критерий «3 решения» измеряет, что **сценарий разбора работает**, а не что пользователь «выбрал кино на вечер».

Векторы **A**, **B**, web, **V0 (вечерний выбор)** — **вне** фазы C; A/B — **parking** до явного подтверждения, что inbox’ом реально пользуются.

## 0.4. Колода (продуктовый target)

| Правило | Значение |
| --- | --- |
| Размер колоды сессии | до **10** лучших кандидатов для **разбора** |
| Действия на карточке | только **смотрел / сохранить / скрыть** (не like/dislike, не «смотреть сейчас») |
| Конец колоды | одна CTA «ещё / обновить» |
| Пул под капотом | не UX; может быть шире 10 |

**Код** размера колоды (сейчас может быть 25) меняется **отдельной задачей**. Здесь зафиксирован только продуктовый target.

## 1. Зачем существует Watchbane

Watchbane — **личный инструмент** (сначала для автора), чтобы:

1. **разобрать** порцию новых / неизвестных рекомендаций;
2. **пополнить** списки watched / saved / hidden без бесконечного скролла.

Это **не** «выбрать один тайтл на вечер», **не** каталог и **не** стриминг.

---

## 2. Job-to-be-done

> Открыл → за разумное время вижу **колоду для разбора** (до 10 карточек с постерами) → за **5 минут** разбираю **≥ 3 карточки** (смотрел / сохранить / скрыть).

**«Разумное время»:** колода с постерами готова без ощущения, что приложение сломано. Ориентир на повторном запуске: **до ~10 секунд**. Холодный старт может быть дольше — с **одним** честным экраном ожидания.

---

## 3. Критерий успеха фазы C (acceptance)

Фаза **C** считается пройденной, если **5 сессий подряд** (минимум 3 в разные дни) выполняются все пункты:

| ID | Критерий | Статус |
| --- | --- | --- |
| S1 | После запуска пользователь сразу на вкладке **Рекомендации**, без лишних шагов | ☐ |
| S2 | До колоды с постерами — разумное время, **один** понятный экран ожидания | ☐ |
| S3 | Колода **конечная** (до 10 карточек, не бесконечный скролл) | ☐ |
| S4 | За ≤ 5 минут разобрано ≥ 3 карточек (смотрел / сохранить / скрыть) | ☐ |
| S5 | Нет технических сообщений, конфликтующих с табами / обрезающихся сверху | ☐ |
| S6 | Daily path **не требует** «Поиск настроек» и детальных фильтров | ☐ |

S4 проверяет **работоспособность inbox**, не факт «начал смотреть кино».

Когда все S1–S6 отмечены — агент закрывает **Фазу C** и фиксирует дату в [журнале прогресса](#журнал-прогресса).

| Приоритет | Вектор | Суть | Когда |

Уникальность сейчас: локальная колода + watched/saved/hidden, а не «магический выбор на вечер» и не NL.

## 5. «Не делать»

Пока активна текущая фаза — **отклонять** задачи, нарушающие этот список:

1. ☐ Не переписывать на web — desktop PyQt остаётся платформой до закрытия фазы C.
2. ☐ Не добавлять LLM / NL-поиск / embeddings — это вектор B.
3. ☐ Не расширять filter algebra (AND/NOT, keywords, runtime, networks) — это вектор A.
4. ☐ Не умножать пресеты, вайб-крутилки и «направления» — их можно **спрятать**, не добавлять.
5. ☐ Не превращать Рекомендации в бесконечный каталог.
6. ☐ Не показывать пользователю техническую цепочку (пул, replenish, discover, фильтрация) — один этап «готовим колоду».
7. ☐ Не требовать daily path через «Поиск настроек».
8. ☐ Не полировать оформление ради красоты, пока не работает happy path (фаза C4 — после C1–C3).
9. ☐ Не проектировать «как будет у остальных» — сначала полезность для одного пользователя.
10. ☐ Не начинать новую большую идею, если inbox-сессия не проходит «≥ 3 разобранных карточки».
11. ☐ Не менять модель действий на like/dislike — только смотрел / сохранить / скрыть.
12. ☐ Не расширять scope за один roadmap ID (без «заодно»).
13. ☐ Не строить сценарий «Сегодня» / кнопку «Смотреть сейчас» / evening shortlist (V0) в фазе C.
14. ☐ Не открывать A/B «потому что C почти закрыт» без явного решения автора.

## 6. Happy path (единственный сценарий v1 — inbox)

Daily path из **6 шагов** — эталон для всех фаз C.  
Чеклист и маппинг UI: [`HAPPY_PATH_INBOX.md`](HAPPY_PATH_INBOX.md).

1. Открыл приложение (TMDb-токен и onboarding уже пройдены).
2. Вкладка **Рекомендации** — главная и первая (**разбор колоды**, не «выбор на вечер»).
3. Видишь колоду **с постерами** (до **10** карточек) или **одно** состояние «готовим колоду».
4. Листаешь **конечную** колоду (не бесконечный скролл).
5. Три действия на карточке: **смотрел / сохранить / скрыть**.
6. Колода кончилась → **одна** кнопка «обновить / ещё варианты».

## 8. Технические границы для агента

| Можно в фазах C | Нельзя в фазах C |
| --- | --- |
| UI/UX Recommendations, empty states, overlay | Переписывание на web |
| Спрятать/упростить controls в daily path | Новые filter dimensions (фаза A) |
| Упростить messaging загрузки | LLM / embeddings (фаза B) |
| Poster flow: один gate, честный прогресс | Менять SQLite schema без крайней нужды |
| Баги, ломающие happy path | ReDeck ranking overhaul «с нуля» |
| Мелкие фиксы выдачи в C3 | Analytics, новые большие фичи «заодно» |
| Смена размера колоды UX → 10 (отдельный ID) | Like/dislike вместо смотрел/сохранить/скрыть |

**Definition of Done** для каждой задачи:

1. Есть diff изменений.
2. Прогнаны релевантные существующие тесты.
3. Ручной прогон по acceptance задачи.
4. Для UI: запуск приложения + скрины + проверка scales **1.0 и 1.25** (см. корневой `AGENTS.md`).
5. Не сломаны Collection / Settings.
6. Commit — **только по запросу пользователя**.
7. Прогресс отмечен в этом файле.

## 9. Как агент отмечает прогресс в этом файле

При закрытии задачи или фазы агент **редактирует этот MD**:

1. Ставит `[x]` у выполненной задачи (C1-01, C2-03, …).
2. Обновляет **«Текущая фаза»** в шапке roadmap (раздел 7).
3. Когда все задачи подфазы закрыты — отмечает **«Статус фазы: пройдена»** и пишет **дату закрытия**.
4. Добавляет строку в [журнал прогресса](#журнал-прогресса): дата, что сделано, что проверено.
5. Обновляет **«Последнее обновление»** в шапке документа.

**Шаблон запроса пользователя:**

```text
Контракт: PRODUCT_ROADMAP_CONTRACT.md
Задача: C2-01
Scope: только то, что в задаче. Не расширять.
После: отметь прогресс в docs/contracts/PRODUCT_ROADMAP_CONTRACT.md
```

**Правило:** одна задача агента = одно изменение поведения (или одна связанная группа UI-состояний). Без «заодно».
